AI Product Ops — Take-home case study

100 apps, one question: can this be an agent toolkit today?

Before Composio builds a toolkit, someone has to find out what auth an app uses, whether a developer can just... get credentials, and whether an API even exists. This is that research, done across 100 real apps, by an agent — with every finding checked, and every mistake shown.

72/100

are self-serve today — a developer can get working credentials for free, no sales call, no approval queue. The other 28 split between partial gates (13, usually a paid-plan wall) and hard gates (15, a contract or partner review). Full breakdown below.

01 — Findings

The patterns, stated plainly

A

Auth is basically solved

97%

use OAuth2, an API key/token, or both. Only Sherlock and Mermaid CLI have no remote auth at all (they're local tools, not services) and DataForSEO is the lone Basic-auth holdout. Whatever toolkit-builder you write only needs to handle two auth shapes to cover almost everything.

B

Infra & PM tools: zero friction

20/20

Developer/Infra and Productivity/PM categories are 100% self-serve — every single app in both. If you want the fastest 20 toolkits to ship, start there and stop researching.

C

Where the gates cluster

7/10

AI/Research/Media apps are gated or mixed (7 of 10) — the category is full of month-old MCP servers whose REST APIs are still Enterprise-only. Finance/Fintech is next (5 of 10): banking-grade underwriting is a real, not bureaucratic, blocker.

D

The #1 blocker isn't "no API"

~10 apps

The most common named blocker is a paid-plan wall — the API exists, works, and is documented, but sits behind a plan tier (Ahrefs, SE Ranking, Squarespace Commerce, Fathom, Grain, GoHighLevel). This is a pricing problem, not a technical one — often the easiest gate to plan around.

E

MCP adoption is already real

32/100

already expose some form of MCP server — official or community. Support & Helpdesk apps (Pylon, Zendesk, Intercom) and Productivity tools are furthest along; Marketing/Ads has the least MCP coverage of any category.

F

True dead ends are rare

6/100

Only 6 apps are genuinely blocked with no near-term path: Pumble (no general API), LinkedIn Ads & PitchBook (hard partner/contract gates), Paygent Connect & Consensus's high-volume tier (opaque access), and Sherlock/NotebookLM-consumer (not services at all).

03 — How it was built

The agent, and where it needed a human

This is not 100 rows typed by hand. A Composio-first research pipeline did the lookups; I (the agent doing this research session) executed its logic live for every row, using real web search and doc-page reads exactly as the script automates.

01 · INPUT

The 100-app list, grouped into 10 categories, each with a name + a docs-site hint.

→

02 · COMPOSIO LOOKUP

Query Composio's own toolkit catalog first (composio-core SDK) — auth scheme, action count, and MCP support are ground truth when an app is already a toolkit.

→

03 · WEB FALLBACK

Miss? Free, keyless search (DuckDuckGo HTML) + doc-page scrape, keyword-matched against auth/gating vocabulary ("OAuth 2.0", "contact sales", "request access"...).

→

04 · STRUCTURED RECORD

Every row gets category, auth, gating, API surface, MCP status, verdict, blocker, evidence URL, and a confidence tag.

In this specific run, step 02 (a live Composio API call) wasn't reachable from the sandboxed research environment — so I substituted real live web search for every uncertain app, which is exactly what step 03 automates. Where this needed a human (or a better tool than search)

      
    
        
  • Ambiguous "self-serve vs. gated" calls. Docs pages that say "contact sales" right next to a working sandbox (Pylon, Otter) need someone to actually read the pricing page and decide which claim wins.
    • 
        
  • Same-name collisions. Searching "Consensus API" surfaces both consensus.app (AI research search — the one on this list) and the unrelated goconsensus.com (sales-enablement software). A keyword match alone would happily merge the two.
    • 
        
  • "Does this count as an MCP" judgment calls. A GitHub repo with 3 stars wrapping an unofficial API (see Otter's community forks) is a very different buildability signal than a provider-hosted OAuth server — the assignment's bar for "existing MCP" needed a human line to be drawn.
    • 
        
  • Dead-end confirmation. For Waterfall.io and Paygent Connect, the first search pass returned nothing conclusive — a human had to try a second, differently-worded query (or, for Paygent, conclude the docs genuinely aren't public) before calling it either way.

04 — Trust, but check

    
Verification: what moved between the first pass and now

    
92 of the 100 rows were populated from documented, stable technical knowledge (e.g. "Stripe uses API keys") rather than a fresh search — that's a reasonable default for mainstream, slow-changing APIs. To find out how reliable that default actually is, 10 of those rows were sampled and re-checked against live docs, cold, as if they'd come out of the pipeline's web-fallback path.

The two misses share a pattern: both were apps that shipped a new access path (an official MCP server) recently. A one-time knowledge snapshot can't know that; a live search catches it immediately. That's the actual argument for the verification loop — not that general knowledge is unreliable, but that it decays fastest exactly where the market is moving fastest.

  






  

    

      

        
Honesty notes

        
92 rows use documented technical knowledge (marked "known" in the matrix); 8 were live-searched this session after being flagged uncertain (marked "live-verified"). All 100 have an evidence link. Two categories worth a callout: Sherlock and Mermaid CLI aren't hosted services at all — including them as "toolkits" would mean shelling out to a local binary, a different integration shape than everything else on the list. Paygent Connect defeated the research entirely — no public docs were found, and that absence is reported as the finding.
